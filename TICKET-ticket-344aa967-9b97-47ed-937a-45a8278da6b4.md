## 1. タスクの目的と背景

- `packages/ice-server` の STUN/TURN 実装が、**Chrome の `RTCPeerConnection` から実際に利用可能**であることを検証する。
- 現状は以下までは確認済み。
  - `packages/ice-server/tests/protocol.test.ts`  
    - STUN Binding 応答
  - `packages/ice-server/tests/turn.test.ts`  
    - TURN 認証、Allocate、CreatePermission、ChannelBind、ChannelData、Node サーバ動作
  - `packages/ice/tests/ice/turn.test.ts`  
    - `werift-ice-server` と `werift` ICE クライアントの接続
- ただし、**Chrome 固有の candidate gathering / TURN 利用 / DataChannel 接続**は未検証。
- そのため、`./e2e` と同系統の **Chrome E2E ハーネス**を `packages/ice-server/chrome-e2e` に持ち、STUN 利用時と TURN 強制利用時の両方で、**Peer 間の双方向 DataChannel 通信**を成功条件にする。

## 2. 実装すべき具体的な機能や変更内容

- `packages/ice-server/chrome-e2e` を新設し、`/e2e` を参考にした独立実行可能なブラウザ E2E 環境を追加する。
  - 想定ファイル例
    - `packages/ice-server/chrome-e2e/package.json`
    - `packages/ice-server/chrome-e2e/tsconfig.json`
    - `packages/ice-server/chrome-e2e/vitest.config.mts`
    - `packages/ice-server/chrome-e2e/run-ci.js`
    - `packages/ice-server/chrome-e2e/stop.js`
    - `packages/ice-server/chrome-e2e/server/main.ts`
    - `packages/ice-server/chrome-e2e/tests/fixture.ts`
    - `packages/ice-server/chrome-e2e/tests/stun.test.ts`
    - `packages/ice-server/chrome-e2e/tests/turn-relay.test.ts`
- Node 側でローカル `NodeStunServer` / `NodeTurnServer` を起動し、Chrome テストへ接続情報を渡す。
- Chrome 側で `RTCPeerConnection` を **2つ**立て、ページ内で SDP / ICE candidate を相互中継して接続を張る。
- 少なくとも以下 2 シナリオを実装する。
  1. **STUN 利用シナリオ**
     - `iceServers: [{ urls: "stun:..." }]`
     - DataChannel の双方向送受信成功
     - STUN サーバへの Binding 要求到達を確認
  2. **TURN 強制利用シナリオ**
     - `iceServers: [{ urls: "turn:...?transport=udp", username, credential }]`
     - `iceTransportPolicy: "relay"`
     - DataChannel の双方向送受信成功
     - `getStats()` 等で relay candidate / selected pair を確認
- `packages/ice-server/package.json` に、必要ならラッパースクリプト（例: `chrome-e2e`）を追加する。
- 実装不具合が見つかった場合は、`packages/ice-server/src/**` を修正し、原因に対する **回帰テスト** を追加する。

## 3. 技術的な実装アプローチを調査し結果を簡潔にまとめる

- 既存 `/e2e` は **Vitest Browser + Playwright + Node サーバ並列起動**の構成になっているため、この実行モデルを流用するのが最も自然。
- ただし今回の対象は `werift` 側のシグナリングではなく `ice-server` 単体なので、`/e2e/server/main.ts` のような大きな WebSocket シグナリング層は不要。
  - **推奨構成**:
    - Node サーバ: STUN/TURN 起動、設定配布、メトリクス収集、停止 API
    - Browser テスト: 2 つの Chrome `RTCPeerConnection` を同一テスト内で直接接続
- TURN は現状の README / 実装上、**UDP/TCP の control transport + UDP relay** が前提なので、Chrome E2E の最初の対象は `turn:...?...transport=udp` が妥当。
- ブラウザ相互運用で重点的に疑うべき実装箇所は以下。
  - STUN Binding 応答  
    - `packages/ice-server/src/protocol.ts`
  - TURN 長期資格情報認証 (`401`, `438`, `REALM`, `NONCE`, `MESSAGE-INTEGRITY`, `FINGERPRINT`)  
    - `packages/ice-server/src/turn/protocol.ts`
  - Allocate / CreatePermission / ChannelBind / Send/Data / ChannelData  
    - `packages/ice-server/src/turn/protocol.ts`
  - TURN フレーミングと 4-byte padding  
    - `packages/ice-server/src/turn/frame.ts`
  - relay address 正規化や bind/advertise address の扱い  
    - `packages/ice-server/src/node/turnServer.ts`
- 特に TURN 失敗時は、RFC 8489 / RFC 8656 と照合しつつ、**Chrome が実際に踏むメソッド列**  
  `Allocate -> CreatePermission / ChannelBind -> ChannelData / Data Indication`  
  をログやメトリクスで追えるようにするのが有効。

## 4. 考慮すべき制約や注意点

- **STUN シナリオでは selected pair が必ず srflx になるとは限らない。**  
  同一ホスト上の 2 Peer では host candidate が選ばれやすいため、STUN 利用確認は **Binding 要求の到達** や **srflx candidate の収集有無**で見る方が安全。
- TURN シナリオは `iceTransportPolicy: "relay"` を必須にして、host/srflx 経路へ逃げないようにする。
- `packages/ice-server/chrome-e2e` はルート `workspaces` に含まれないため、**独立した `package.json` と実行スクリプト**が必要。
- テストコードはリポジトリ規約に従い、**Arrange/Act/Assert** を意識し、共有セットアップは helper に寄せ、Act/Assert には必要に応じて **日本語コメント** を付ける。
- CI/ローカル双方でポート競合を避けるため、**動的ポート確保**が望ましい。
- 実装調査で不具合が見つかった場合、表面的に回避せず、`ice-server` 側の RFC 準拠に沿って修正する。

## 5. 完了条件

- `packages/ice-server/chrome-e2e` に Chrome E2E 実行基盤が追加され、単独実行できる。
- Chrome Headless で以下が再現性を持って通る。
  1. **STUN 利用時**に 2 つの `RTCPeerConnection` が接続し、DataChannel で双方向通信できる
  2. **TURN 強制利用時**に relay candidate 経由で接続し、DataChannel で双方向通信できる
- STUN/TURN の利用事実を、candidate/statistics またはサーバ側メトリクスで確認できる。
- もし途中で `ice-server` 実装の不具合が見つかった場合、その修正と回帰テストが追加されている。
- 既存の `packages/ice-server` 単体テストと型チェックを壊していない。
