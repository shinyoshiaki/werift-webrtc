

## 1. タスクの目的と背景

- `packages/sctp/README.md` のサンプルが `server.onReceive = ...` / `client.onReceive = ...` になっている一方、実装側 `packages/sctp/src/sctp.ts` では `onReceive` は `readonly onReceive = new Event<[number, number, Buffer]>()` です。  
- 実際の正しい利用パターンは `onReceive.subscribe(...)` で、`packages/sctp/examples/ping.ts` / `pong.ts` / `tests/udp.test.ts` でもその形が使われています。  
- つまり「公開ドキュメントと実APIの不一致」が根本原因で、TypeScript利用者が README 通りに書くと `TS2540` で破綻します。

## 2. 実装すべき具体的な機能や変更内容

- **必須対応（本タスクの主対象）**
  - `packages/sctp/README.md` の `onReceive` 使用例を `onReceive.subscribe((streamId, ppId, data) => ...)` に修正。
  - サーバー/クライアント両方の例を同じ流儀に統一。
- **推奨対応（再発防止）**
  - README のコードを型チェック対象に含める（最小構成の doc-snippet test か、既存 `examples` 参照への置き換え）。
  - API利用方針を明記（`Event` は代入ではなく `subscribe/once/asPromise` を使う）。
- **関連不整合（別タスク化推奨）**
  - README の `import { createUdpTransport } from "werift-sctp"` は現状 `packages/sctp/src/index.ts` で未exportのため成立しないので、同issue系として別スコープ管理が妥当です。

## 3. 技術的な実装アプローチ（調査結果サマリ）

- **方針A（最小・安全）**: ドキュメント修正のみ。  
  - 既存実装・既存テスト・他パッケージ慣例（`readonly Event + subscribe`）と完全整合し、破壊的変更なし。
- **方針B（互換レイヤ追加）**: `onReceive` 代入を受ける setter を追加。  
  - 既存 `Event` 設計とズレやすく、他の `readonly Event` 群との一貫性を崩すため非推奨。
- **結論**: 本件はAPI変更ではなく「README誤記修正」が本筋で、必要なら docs/snippet の型検証を追加して品質担保するのが最適です。

## 4. 考慮すべき制約や注意点

- `werift` 全体でイベント公開は `readonly Event` が標準パターン（`webrtc` 側も同様）なので、`onReceive` を writable callback に変えるのは避けるべきです。  
- ドキュメント修正時、サンプル内の import/export 整合（`createUdpTransport` 問題）も同時確認しないと「READMEは直ったが動かない」が残ります。  
- TypeScriptユーザー向けには「代入不可・購読APIを使う」点を明示し、`streamId / ppId / data` の引数意味を併記すると誤用が減ります。
