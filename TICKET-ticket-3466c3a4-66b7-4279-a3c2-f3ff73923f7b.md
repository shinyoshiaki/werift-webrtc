**詳細化結果:** 調査の結果、報告対象の `packages/rtp/src/srtp/cipher/ctr.ts` だけでなく、既定で優先される `packages/rtp/src/srtp/cipher/gcm.ts` にも受信時の認証タグ検証漏れがあります。したがって、この修正タスクは **AES-CM/HMAC-SHA1-80 と AEAD AES-128-GCM の両プロファイルを対象に、SRTP/SRTCP 受信時の完全性検証を必須化するタスク** として詳細化するのが妥当です。

## 1. タスクの目的と背景

SRTP/SRTCP の受信処理で認証タグを検証せずに復号・受理しているため、**完全性保証が失われている**。その結果、攻撃者が SRTP 鍵を取得できる条件下では、偽造 RTP/RTCP パケットの注入が成立しうる。

コード上の確認ポイントは以下です。

- `packages/rtp/src/srtp/cipher/ctr.ts`
  - `decryptRtp()` は末尾の auth tag を切り落とすだけで検証していない
  - `decryptRTCP()` は `actualTag` を取り出すが `// todo impl compare` のまま未検証
  - `decryptRTCP()` は `isEncrypted === 0` の場合、認証確認前に return する
- `packages/rtp/src/srtp/cipher/gcm.ts`
  - `decryptRtp()` は `setAuthTag()` / `setAAD()` / `final()` がなく、GCM タグ検証が行われていない
  - `decryptRTCP()` も `setAuthTag()` / `final()` がない
- `packages/webrtc/src/secureTransportManager.ts`
  - 既定の SRTP プロファイル優先順は `SRTP_AEAD_AES_128_GCM` → `SRTP_AES128_CM_HMAC_SHA1_80`

加えて、既存テストは `packages/rtp/tests/srtp/**` に正系の往復テストや既知ベクタ検証はあるものの、**改ざんパケットを拒否する負系テストが存在しない**。

## 2. 実装すべき具体的な機能や変更内容

1. **SRTP(AES-CM/HMAC-SHA1-80) の受信認証を実装する**
   - `ctr.ts` の `decryptRtp()` で、受信パケット末尾の auth tag を再計算結果と比較する
   - 一致しない場合は復号結果を返さず失敗扱いにする

2. **SRTCP(AES-CM/HMAC-SHA1-80) の受信認証を実装する**
   - `ctr.ts` の `decryptRTCP()` で、SRTCP index/E-bit を含む認証対象に対して HMAC-SHA1-80 を再計算し比較する
   - `isEncrypted === 0` の場合でも auth tag 検証は必須にする

3. **SRTP/SRTCP(AEAD AES-128-GCM) のタグ検証を正しく有効化する**
   - `gcm.ts` の `decryptRtp()` で `setAAD(header)`, `setAuthTag(tag)`, `final()` を使ってタグ検証を行う
   - `gcm.ts` の `decryptRTCP()` でも `setAAD(aad)`, `setAuthTag(tag)`, `final()` を使う

4. **認証失敗時に状態を壊さないようにする**
   - `packages/rtp/src/srtp/context/srtp.ts` は `decryptRtp()` 内で先に `updateRolloverCount()` しているため、認証失敗でも ROC 状態が進む可能性がある
   - **検証用の ROC 推定** と **成功後の状態反映** を分離する

5. **回帰テストを追加する**
   - 既存の `packages/rtp/tests/srtp/context/srtp.test.ts`
   - `packages/rtp/tests/srtp/context/srtcp.test.ts`
   - `packages/rtp/tests/srtp/cipher/gcm.test.ts`
   - 必要に応じて `packages/rtp/tests/srtp/srtp.test.ts`, `srtcp.test.ts`
   - 追加すべき観点:
     - auth tag 改ざん
     - 暗号文改ざん
     - GCM の AAD 対象ヘッダ改ざん
     - SRTCP index/E-bit 改ざん
     - 認証失敗後に正常パケットを継続受信できること

6. **必要なら WebRTC 受信境界で明示的に破棄する**
   - `packages/webrtc/src/transport/dtls.ts` では `this.srtp.decrypt(data)` / `this.srtcp.decrypt(data)` の直前に専用 catch がない
   - 現状でも ICE 側で例外は拾われるが、**無効 SRTP/SRTCP を明示的に破棄してログする** ほうが影響範囲を限定しやすい

## 3. 技術的な実装アプローチを調査し結果を簡潔にまとめる

- **AES-CM/HMAC-SHA1-80**
  - 送信側は既に `generateSrtpAuthTag()` / `generateSrtcpAuthTag()` を使ってタグ生成している
  - 受信側では同じ計算を使って期待タグを再生成し、`crypto.timingSafeEqual` で比較するのが最小変更で安全
  - SRTP は ROC が HMAC 入力に入るため、**ROC を仮計算して検証し、成功時のみ state に反映** する設計が必要

- **AEAD AES-128-GCM**
  - Node.js の GCM は `decipher.update()` だけでは検証完了にならず、`setAuthTag()` と `final()` が必要
  - RTP はヘッダを AAD として与える必要があるため、復号時も送信時と対になる AAD 設定が必要
  - RTCP も同様に既存 `rtcpAdditionalAuthenticatedData()` を AAD に使える

- **API/エラー契約**
  - 既存 `SrtpSession.decrypt()` / `SrtcpSession.decrypt()` は平文を返すだけなので、認証失敗時は `Error` を送出する形が自然
  - 呼び出し元でパケット破棄に繋げれば、偽造パケットを上位層へ通さずに済む

## 4. 考慮すべき制約や注意点

- **修正対象は `ctr.ts` だけに限定しない**
  - 実運用で優先されるのは GCM のため、`gcm.ts` の未検証も同時に塞がないと不完全
- **ROC 更新タイミングに注意**
  - 認証前に `updateRolloverCount()` すると、不正パケットで状態が変化しうる
- **SRTCP は非暗号化でも認証が必要**
  - `isEncrypted === 0` を理由に auth check を飛ばさない
- **公開 API 互換性**
  - 失敗時に例外が増えるため、受信側で握って破棄する責務をどこに置くかを明確にする
- **既存ベクタを壊さない**
  - `packages/rtp/tests/srtp/context/context.test.ts` などの鍵導出・ROC テストはそのまま通る必要がある
- **比較は定数時間で行う**
  - HMAC タグ比較は通常の `equals` より `timingSafeEqual` を優先する

## 5. 完了条件

- `SRTP_AES128_CM_HMAC_SHA1_80 (0x0001)` と `SRTP_AEAD_AES_128_GCM (0x0007)` の両方で、**受信時に認証タグが必ず検証される**
- auth tag / 暗号文 / AAD(ヘッダ) / SRTCP index を改ざんしたパケットが **復号成功しない**
- 認証失敗したパケットが **RTP/RTCP として上位処理へ渡らない**
- 認証失敗時に **SRTP の ROC/state が不正に更新されない**
- 既存の正系 SRTP/SRTCP テストが維持され、追加した負系テストが通る

## タスク化するならこの書き方が適切

**タスク名**  
SRTP/SRTCP 受信時の認証タグ検証を実装し、改ざんパケットを拒否する

**タスク説明**  
`packages/rtp` の SRTP/SRTCP 復号処理では、AES-CM/HMAC-SHA1-80 および AEAD AES-128-GCM の双方で受信時認証が未実装または不完全であり、改ざんパケットを受理しうる。`ctr.ts` と `gcm.ts` の受信処理に認証タグ検証を追加し、認証失敗時は平文を返さずパケットを破棄すること。あわせて `SrtpContext.decryptRtp()` の ROC 更新を「検証前の推定」と「成功後の反映」に分離し、不正パケットで受信状態が壊れないようにする。`packages/rtp/tests/srtp/**` に正系・負系の回帰テストを追加し、必要に応じて `packages/webrtc/src/transport/dtls.ts` 側でも無効 SRTP/SRTCP を明示的に破棄する。
