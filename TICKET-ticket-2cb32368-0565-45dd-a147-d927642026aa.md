# 詳細化案: E2E終了時にgstreamerプロセスを残さない

## 1. 目的と背景
E2Eの mediachannel 系テストでは、サーバー側で `gst-launch-1.0` を `spawn` して映像/音声ソースを作っています。現状は各テストの `done` 経由で個別に `SIGINT` を送る前提ですが、**テスト失敗・タイムアウト・ブラウザ切断・CI中断**が起きると、その終了処理が通らず gstreamer が orphan 化する可能性があります。

特に現状は、`e2e/package.json` の `chrome:prod` が `vitest ... && node stop.js` になっており、**成功時しか stop 処理が走りません**。また `e2e/server/main.ts` の `/stop` は `process.exit()` するだけで、子プロセスの回収をしていません。

## 2. 実装すべき具体的な変更
- E2Eサーバー側で `gst-launch-1.0` の起動を一元管理する
- すべての gstreamer 子プロセスを登録し、終了時にまとめて停止する
- `/stop` 受信時だけでなく、`SIGINT` / `SIGTERM` / 異常終了時も同じ cleanup を通す
- `chrome:prod` / `run-ci.js` の終了経路を見直し、**テスト成功時だけでなく失敗時も cleanup が必ず実行される**ようにする
- 既存の個別 `process.kill("SIGINT")` は共通 helper に寄せる
- 可能なら `pkill -9 gst-launch` のような全体殺しは依存しない構成にする

## 3. 技術的な実装アプローチ（調査結果）
調査した範囲では、gstreamer を起動しているのは主に以下です。

- `e2e/server/handler/mediachannel/addTrack.ts`
- `e2e/server/handler/mediachannel/red.ts`
- `e2e/server/handler/mediachannel/rtx.ts`
- `e2e/server/handler/mediachannel/removeTrack.ts`

また、`removeTrack.ts` は `trackSources` の `Map` で cleanup を持っていますが、**サーバー全体の終了時回収はありません**。

サーバーの終了経路はこうなっています。

- `e2e/server/main.ts` の `/stop` は `res.send(); process.exit();`
- `e2e/package.json` の `chrome:prod` は `... && node stop.js`
- `e2e/run-ci.js` は `run-p serverScript chrome:prod` を起動

つまり、**テスト失敗時に stop.js が実行されず、gst-launch が残る構造**です。  
このため、共通 cleanup registry を作り、サーバー終了前に「登録済み child process を順に停止 → exit」という順に変えるのが筋です。

## 4. 考慮すべき制約・注意点
- cleanup は **冪等** にする（複数回呼ばれても壊れない）
- `SIGINT` だけで止まらない場合のフォールバックを用意する
- テストごとに複数の gst-launch が立つケースを扱えること
- `pkill -9 gst-launch` のようなグローバル削除は、他ジョブや他利用者を巻き込むので最終手段に留める
- `process.exit()` を先に呼ぶと cleanup が中断されるので、**exit 前に await で停止完了を待つ**
- 既存の E2E テストの意味論は壊さないこと

## 5. 完了条件
- E2E が **成功しても失敗しても**、このテスト実行で起動した `gst-launch-1.0` が残らない
- `/stop` / `SIGINT` / `SIGTERM` / 例外終了のいずれでも cleanup が通る
- CI で手動 `pkill -9 gst-launch` に頼らなくてよい
- 既存の E2E テストが従来どおり通る
