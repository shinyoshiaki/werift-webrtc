

# タスク詳細化: WPTで一部成功しているファイルを列挙し、成功数を増やす互換性改善

## 1. タスクの目的と背景

`packages/webrtc` には upstream Web Platform Tests を直接実行する WPT ランナーがあり、`third_party/wpt/webrtc` 配下の **214 ターゲット**を実行できます。現状の実測では **1066 subtest 中 253 PASS / 781 FAIL / 32 TIMEOUT** で、**少なくとも 1 件 PASS している対象は 65 ファイル（68 file+variant）**あります。

このため本タスクの本質は、**完全未対応領域を広く触るより、すでに部分的に互換性があるファイル群を優先して PASS 数を押し上げる**ことです。  
なお現在の `coverage/webrtc-wpt/results.md` は失敗/タイムアウト中心の要約で、**「1件以上成功しているファイル一覧」は標準出力されません**。そのため、まず可視化を足し、その上で実装改善を進めるのが自然です。

## 2. 実装すべき具体的な機能や変更内容

1. **WPT結果の可視化追加**
   - `coverage/webrtc-wpt/results.json` を集計し、**PASS が 1 件以上ある file(+variant) 一覧**を出せるようにする。
   - 出力項目は最低でも `file`, `variant`, `pass`, `fail`, `timeout`。
   - 既存 `results.md` 拡張か、専用の list/summarize コマンド追加が妥当。

2. **互換性改善の優先実装**
   - **RTCPeerConnection の state machine / rollback / parameterless SLD**
   - **track / transceiver / ontrack / stream id の整合**
   - **addIceCandidate の入力検証と candidate 適用先解決**
   - **createDataChannel オプションの Web IDL 風 coercion・range check**
   - **例外の型と名前のブラウザ互換化**（`Error` ではなく `TypeError` / `DOMException`）
   - **必要なら `RTCRtpSender.setStreams()`、`getParameters()` / `setParameters()` の不足補完**

3. **関連テストの追加・更新**
   - 既存の package test に、WPT failure を再現する小さい回帰テストを足す。
   - 候補: `tests/integrate/peerConnection.test.ts`, `tests/integrate/trickle.test.ts`, `tests/datachannel/send.test.ts`, `tests/integrate/getStats.test.ts`

## 3. 技術的な実装アプローチ

- 実行経路は `packages/webrtc/tools/wpt-runner/run.ts` → `runner.ts` で、結果は `coverage/webrtc-wpt/results.json` に保存されます。
- `discoverWptTargets()` は `third_party/wpt/webrtc` 配下の HTML を自動発見するため、**対象選定の source of truth は `allowlist.json` ではなく実行結果 JSON**です。
- まず **results.json を file+variant 単位で group by し、`pass > 0` を抽出する集計処理**を追加します。
- その後、PASS 比率が高い領域から順に実装修正します。

### 主な修正先

- `src/peerConnection.ts`
  - `setLocalDescription`, `setRemoteDescription`, `addIceCandidate`, `createDataChannel`, constructor/config validation
- `src/sdpManager.ts`
  - pending/current description と rollback
- `src/secureTransportManager.ts`
  - ICE candidate の解決・検証
- `src/transceiverManager.ts`
  - transceiver 順序、`ontrack.streams`、remote stream id
- `src/media/rtpSender.ts`
  - `setStreams`, `getParameters`, `setParameters`, sendEncodings 周辺
- `src/sctpManager.ts` / `src/dataChannel.ts`
  - DataChannel 初期化オプションの coercion / null / range / error semantics
- `src/media/stats*`
  - `mandatory-getStats` 対応

### 推奨実施順

1. 可視化追加  
2. `removeTrack` / `addTrack` / constructor error semantics  
3. `ontrack` / remote stream id / transceiver order  
4. `parameterless SLD` / `SRD rollback`  
5. `addIceCandidate`  
6. `DataChannel` option coercion  
7. `getParameters` / `setParameters` / stats 拡張

## 4. 考慮すべき制約や注意点

- WPT は `third_party/wpt` submodule 前提。
- **`allowlist.json` の `enabled` / `subtests` は現状 runner で実行制御に使われていません。**
- WPT は **例外クラス・例外名・IDL属性の位置**まで厳密に見ます。`Error` と `TypeError`、`DOMException("InvalidStateError")` の差で落ちます。
- `npm run wpt` の成功は「baseline regression がない」意味であり、**PASS 数が増えたことの証明には `results.json` の比較が必要**です。
- 公開 API や互換性メモを変える場合は `packages/webrtc/README.md` も更新対象です。
- baseline/coverage を意図的に広げる場合だけ `wpt/baseline.json` や `wpt/coverage-baseline.json` を更新します。
