## 1. タスクの目的と背景

- 既存コードベースには、親チケットで求められていた基本構成がすでにあります。  
  - `examples/turn-loopback/server/main.ts`  
    - HTTPS `POST /session` / `PUT /session`
    - server-offer / client-answer
    - `createDataChannel("loopback")`
    - セッション単位の一意な `username` / `password`
    - DataChannel echo
    - HTTPS と TURN/TLS の同一ポート多重化
  - `examples/turn-loopback/src/App.tsx`  
    - React SPA
    - `turns:` URI と一時資格情報を受けて browser `RTCPeerConnection` を作成
    - `iceTransportPolicy: "relay"` の client-only 強制
    - ICE gathering 完了後の `PUT /session`
  - `packages/ice-server/src/node/turnServer.ts`  
    - `attachTlsSocket()` と `tls.external` がすでにあり、外部 TLS listener から TURN へ socket handoff 可能
  - `packages/ice-server/tests/turn.test.ts`  
    - external TLS socket handoff のテストあり
- そのため今回の目的は、**既存 example を “ローカル開発用サンプル” から “単一アドレスでそのまま使える配布可能サンプル” に引き上げること**です。
- 特に、現状は  
  - client dev 起動と server 起動が分離されている  
  - server がビルド済み client を配信しない  
  - Dockerfile がない  
  - example 専用の Playwright E2E がない  
  ため、この部分を補完するのが追加タスクの本体です。

## 2. 実装すべき具体的な機能や変更内容

| 領域 | 現状 | 追加で必要 |
| --- | --- | --- |
| server signaling/TURN | ほぼ実装済み | static file 配信、E2E 用 stop 導線、公開ホスト解決の見直し |
| client SPA | ほぼ実装済み | dev/prod で接続先の挙動を分離 |
| Docker | 未実装 | example 用 Dockerfile と起動手順 |
| E2E | 未実装 | Playwright で単一アドレス動作と複数セッション確認 |

### サーバー側
- `examples/turn-loopback/server/main.ts` に、**Vite build 済みファイルの静的配信**を追加する。
  - `GET /` → `dist/index.html`
  - `GET /assets/*` → Vite 出力物
  - API (`/session`, `/health`) は現状維持
  - SPA なので `GET` の fallback を `index.html` に寄せる設計が安全
- **build 済み client を server 自身が配信**し、`https://<host>:<port>/` だけで利用可能にする。
- `turnUrl` のホスト部は、現状の固定 `TURN_LOOPBACK_PUBLIC_HOST` 依存だけではなく、**実アクセス先と一致する値**を返せるようにする。
  - 優先順は `TURN_LOOPBACK_PUBLIC_HOST` 明示指定 → request の `Host` ヘッダ由来、が自然
  - Docker/リモートアクセス時に `127.0.0.1` 固定返却だと破綻する
- E2E ハーネス向けに、必要なら `PUT /stop` を追加する。
  - `packages/ice-server/chrome-e2e/server/main.ts` と同じ停止パターンを流用できる

### クライアント側
- `examples/turn-loopback/src/App.tsx` の signaling base URL を **実行モードで分岐**する。
  - **dev**: 任意アドレスを指定可能
  - **build/prod**: `window.location.origin` を使い、ホスト中の同一 server を使う
- 現状の入力 UI は dev 要件には合っているため、production では
  - 非表示
  - read-only
  - もしくは初期値を same-origin 固定  
  のいずれかで「ビルド時は同一 server を使う」を明確化するのがよい
- echo 結果表示、`ondatachannel`、ICE gathering 完了待ちは現状実装を踏襲する

### Docker
- `examples/turn-loopback/Dockerfile` を追加する
- この example の server は `packages/*/src` を直接参照しているため、**Docker build context は example 単体ではなく repo root 前提**にする
  - 例: `docker build -f examples/turn-loopback/Dockerfile .`
- 画像起動後に `https://<host>:<port>/` にアクセスすれば、server が配信する client からそのまま echo まで完結する構成にする
- 証明書は既存 TURN E2E と同じ `packages/dtls/assets/cert.pem`, `key.pem` をデフォルト利用しつつ、環境変数上書きを維持する

### E2E
- example 専用の Playwright ハーネスを追加する
  - 配置は `examples/turn-loopback/chrome-e2e` が既存パターンに近い
- 最低限必要なシナリオ
  1. 単一アドレス配信された SPA を開いて echo 成功
  2. 2 ページ以上で同時接続し、各セッションが独立成功
  3. dev モード client が任意の signaling/echo server URL を使える
- Chrome/Chromium 起動時に証明書制限を緩和する
  - `--ignore-certificate-errors`
  - `--allow-insecure-localhost`

## 3. 技術的な実装アプローチを調査し結果を簡潔にまとめる

### 再利用すべき既存実装
- **同一ポート多重化**  
  `examples/turn-loopback/server/main.ts` の `routeSecureSocket()` が、TLS 復号後の先頭データで HTTP と TURN を振り分けており、この方式をそのまま使える
- **TURN external socket handoff**  
  `packages/ice-server/src/node/turnServer.ts` の `attachTlsSocket()` / `tls.external` はすでに存在し、追加の `packages/ice-server` 改修は原則不要
- **証明書制限緩和付き Playwright**  
  `packages/ice-server/chrome-e2e/vitest.config.mts` と `e2e/vitest.config.mts` の Chromium launch args をそのまま踏襲できる
- **動的ポートの E2E 起動**  
  `packages/ice-server/chrome-e2e/run-ci.js` の free port 確保 + health check + stop API の流れを流用できる

### 推奨方針
- **server の static hosting は現行の `node:http` ベースに寄せる**
  - この example は Express 依存なしで完結しているので、静的ファイル配信も同じファイルで閉じるのが自然
- **client の接続先は `import.meta.env.DEV` で分岐**
  - dev: `VITE_SIGNALING_BASE_URL` または手入力
  - prod: `window.location.origin`
- **Docker は “client を build して server が配信する” 方式**
  - server 自体は現状どおり `tsx` 実行でもよい
  - server が workspace source を直接読むため、無理に standalone JS 化しない方が変更範囲が小さい
- **E2E は example ローカルに閉じる**
  - root `e2e` に混ぜるより、`packages/ice-server/chrome-e2e` と同様の package-local harness が保守しやすい

## 4. 考慮すべき制約や注意点

- **server peer は TURN client にしない**
  - これは現状 `examples/turn-loopback/server/main.ts` で守られているので、static hosting や Docker 化で崩さない
- **production で `127.0.0.1` 固定返却は不可**
  - 単一アドレス運用と Docker 要件を満たすには、`turns:` URI の host が実アクセス先と一致している必要がある
- **self-signed cert 前提では browser 実機利用は厳しい**
  - E2E では Playwright 側で緩和できるが、実運用用途では有効証明書か信頼済み証明書が必要
- **Vite build 産物はハッシュ付き asset 名**
  - server 側は `dist/assets/*` をそのまま返せる必要がある
- **example 単体 Docker context では不足しうる**
  - `packages/webrtc/src`, `packages/ice-server/src`, `packages/dtls/assets` を参照するため、repo root context 前提で設計する必要がある
- **複数セッション確認は UI だけでなく E2E で担保する**
  - 現状 server の session map はあるが、追加要件の完了条件としては browser-level 実証が必要

## 5. 完了条件

1. `https://<host>:<port>/` で server 配信の SPA にアクセスできる  
2. build 済み client が同一 origin の server に対して `POST /session` / `PUT /session` を行う  
3. 返却される `turns:` URI が実アクセス先と整合する  
4. dev モードでは client から任意アドレスの echo server を指定できる  
5. server 側 peer に TURN 設定や `iceTransportPolicy: "relay"` を入れない方針が維持される  
6. シグナリング完了後に DataChannel echo が成功し、画面に反映される  
7. 複数ページ/複数タブで同時接続してもセッションが衝突しない  
8. `examples/turn-loopback/Dockerfile` から起動したコンテナだけでアプリ全体が利用できる  
9. Playwright E2E で単一アドレス配信と複数セッション成功を確認できる  
10. E2E では既存 TURN E2E 証明書を使い、Chromium の証明書制限緩和設定が入っている  
11. README に dev 起動、build 利用、Docker 起動、E2E 実行手順が揃っている  

以上を踏まえると、このタスクは**新規 example 作成ではなく、既存 `examples/turn-loopback` を本番寄りの配布形に仕上げる追補タスク**として扱うのが最も自然です。
