## 1. タスクの目的と背景

`packages/ice` には既に STUN 関連の基盤があります。

- `packages/ice/src/stun/message.ts` / `attributes.ts` / `const.ts`: STUN メッセージの表現と属性
- `packages/ice/src/stun/protocol.ts`: UDP ベースの STUN クライアント寄り実装
- `packages/ice/src/iceBase.ts` の `serverReflexiveCandidate()`: STUN サーバーに Binding Request を送って srflx candidate を取得
- `packages/ice/src/ice.ts` の `Connection.ensureProtocol()`: ICE 用の Binding Request に対する応答処理

一方で、**STUN サーバーとして再利用できる独立パッケージ**はなく、**プロトコル処理とネットワーク I/O が分離されていません**。また `packages/ice/tests/ice/ice.test.ts` では `stun.l.google.com` を使うテストがあり、外部依存です。

このタスクの目的は次の 3 点です。

1. `packages/ice-server` に **RFC 8489 ベースの STUN サーバー実装**を追加する  
2. **Sans-IO** でプロトコル層と Node.js のネットワーク層を分離し、利用者が任意のサーバー実装を作れるようにする  
3. `packages/ice` の 2 Peer が、その STUN サーバーを使って candidate gathering と P2P 通信まで通ることを確認する

## 2. 実装すべき具体的な機能や変更内容

### 追加するパッケージ

`packages/ice-server` を新規追加し、既存パッケージと同じ構成で持たせるのが自然です。

- `package.json`
- `tsconfig.json` / `tsconfig.production.json`
- `src/index.ts`
- `tests/`

パッケージ名は既存命名に合わせると `werift-ice-server` が有力です。

### 実装する機能

#### 1. Sans-IO の STUN プロトコル層
ネットワークに依存しない pure な層を持たせます。

想定責務:

- 受信 datagram の STUN 判定
- STUN Message の parse / serialize
- Binding Request / Binding Indication の処理
- Success Response / Error Response の生成
- `XOR-MAPPED-ADDRESS` の付与
- `ERROR-CODE` / `UNKNOWN-ATTRIBUTES` の生成
- `SOFTWARE` / `FINGERPRINT` の付与ポリシー
- 将来の認証拡張用フック

API は例えば以下のような方向が適しています。

- `handleDatagram(input) => Action[]`
- `input`: `data`, `remoteAddress`, `localAddress`, `transport`
- `Action`: `send(responseBytes, addr)` / `ignore`

#### 2. Node.js の参照サーバー実装
Node.js 上で動く UDP サーバーを用意します。

責務:

- UDP ソケット bind / close
- 受信パケットを pure なプロトコル層へ渡す
- 返ってきた action をそのまま send する
- 設定可能項目: `host`, `port`, `software`, 認証フックなど

実装は `node:dgram` 直書きでもよいですが、既存の `packages/common/src/transport.ts` の `UdpTransport` を使うと repo の流儀に合います。

#### 3. 既存 STUN 実装の再利用または抽出
現状の `packages/ice/src/stun/*` は再利用候補ですが、そのままではサーバー用途に不足があります。

主な不足:

- 厳密な STUN ヘッダ検証が弱い
- unknown comprehension-required attribute を保持できない
- `UNKNOWN-ATTRIBUTES` 属性が未実装
- `MESSAGE-INTEGRITY-SHA256` / `USERHASH` など RFC 8489 の拡張属性は未実装
- `StunProtocol` が UDP ソケット生成まで持っており Sans-IO ではない

そのため、**`packages/ice/src/stun/*` の汎用部分を `packages/ice-server` 側に寄せて、`packages/ice` から再利用させる**のが理想です。

#### 4. `packages/ice` との統合テスト
既存 API だけで統合可能です。`Connection` は既に `stunServer?: Address` を受け取れます。

追加/更新するテスト:

- ローカル STUN サーバー起動
- `new Connection(true, { stunServer: [host, port] })`
- `new Connection(false, { stunServer: [host, port] })`
- `inviteAccept(a, b)` → `connect()` → 双方向 `send()`
- candidate gathering が通ること
- 必要なら `srflx` candidate が追加されること

## 3. 技術的な実装アプローチを調査し結果を簡潔にまとめる

### 現状コードから見た実装ポイント

| ファイル | 現状の役割 | 今回の示唆 |
|---|---|---|
| `packages/ice/src/stun/message.ts` | STUN メッセージ parse/serialize | codec の出発点として使える |
| `packages/ice/src/stun/attributes.ts` | STUN 属性定義 | 未対応属性の追加が必要 |
| `packages/ice/src/stun/protocol.ts` | UDP STUN client | Sans-IO 化の対象ではなく、分離設計の参考 |
| `packages/ice/src/iceBase.ts` | `serverReflexiveCandidate()` | 新サーバーとの疎通確認に直結 |
| `packages/ice/src/ice.ts` | ICE peer 本体、Binding 応答も内包 | STUN サーバーとは別責務として切り出すべき |
| `packages/common/src/transport.ts` | UDP/TCP transport 抽象 | Node 参照サーバー層で再利用しやすい |
| `packages/ice/tests/ice/ice.test.ts` | P2P / STUN 利用テスト | ローカル STUN サーバーを使う統合確認先 |
| `pion/turn/server_config.go` | transport と server config の分離 | Sans-IO + 参照実装の API 設計の参考になる |

### 推奨アプローチ

#### A. 実装スコープはまず **Binding over UDP** を主軸にする
`packages/ice` が現在必要としているのは **unauthenticated Binding request/response over UDP** です。  
`Connection.gatherCandidates()` もその前提で動いています。

したがって、最初の完了形は:

- Binding Request を受け取る
- `XOR-MAPPED-ADDRESS` を返す
- unknown required attributes には 420 を返す
- Binding Indication は無応答
- 必要に応じて `SOFTWARE` / `FINGERPRINT` を付ける

この形が最も repo の現在値に合っています。

#### B. サーバーは stateless 寄りにできる
RFC 8489 上、Binding は idempotent なので、**参照サーバーは transaction state を強く持たずに再計算型で応答**できます。  
Sans-IO の pure handler と相性がよいです。

#### C. `packages/ice` 側 API はほぼ既存のままでよい
`Connection` はすでに `stunServer` を持っているため、`packages/ice` 本体への変更は大きくありません。  
主な変更点は **テストと依存関係の配線**です。

#### D. テストは外部 STUN 依存を減らす方向がよい
今の Google STUN 依存テストは不安定要因になり得るため、**ローカル STUN サーバーによる統合テストを追加し、できれば主要確認をそちらへ寄せる**のがよいです。

## 4. 考慮すべき制約や注意点

### 1. 「RFC 8489 準拠」のスコープを現実的に切る必要がある
RFC 8489 全体には以下も含まれます。

- UDP 以外の TCP / TLS / DTLS
- short-term / long-term authentication
- `MESSAGE-INTEGRITY-SHA256`
- `USERHASH`
- `ALTERNATE-SERVER` など

ただし、**現在の repo が実際に必要としているのは UDP の Binding server** です。  
このタスクでは **Basic Server Behavior + Binding over UDP** を first milestone とし、他は拡張可能な設計に留めるのが妥当です。

### 2. 現在の `packages/ice` は STUN を IPv4 gathering でしか実質使っていない
`packages/ice/src/ice.ts` の `gatherCandidates()` では、STUN 問い合わせは IPv4 host candidate に対してのみ行っています。  
そのため、**今回の完了判定は IPv4 経路で十分**です。

### 3. ローカル STUN サーバー試験では「NAT 変換された reflexive address」にならない場合がある
同一ホスト/同一 LAN 内でテストすると、STUN サーバーが見た送信元が host candidate と同じになることがあります。  
そのため統合テストでは **「実 public IP を得ること」ではなく、「STUN サーバーを使って candidate gathering と P2P 通信が成立すること」** を確認対象にするのが安全です。

### 4. STUN codec の二重実装は避ける
`packages/ice` と `packages/ice-server` に STUN 定義が二重化すると、将来的に属性対応や validation がズレます。  
**可能なら共通 core に寄せる**前提で進めるべきです。

### 5. 新規 package は monorepo 規約に合わせる
root は `packages/*` を workspace に含んでいるため、新規 package は build/test/type script を揃える必要があります。  
既存 package と同様の script 構成にしておくのが前提です。

## 5. 完了条件

以下を満たせば完了と見なせます。

1. `packages/ice-server` が workspace package として追加されている  
2. パッケージが **Sans-IO のプロトコル層**と **Node.js の参照 UDP サーバー層**を分けて公開している  
3. 参照サーバーが Binding Request に対して RFC 8489 の基本動作で応答できる  
   - `XOR-MAPPED-ADDRESS` を返す  
   - Binding Indication には応答しない  
   - unknown comprehension-required attribute には 420 + `UNKNOWN-ATTRIBUTES` を返せる  
4. `packages/ice` の 2 Peer が、その STUN サーバーを `stunServer` として使い、candidate gathering → `connect()` → 双方向データ送信まで成功する自動テストがある  
5. 既存 monorepo の build / type / test フローに新 package が組み込まれている  
6. 必要最低限の README または package 公開 API 説明がある
